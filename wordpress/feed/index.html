awesome</a>. Although the official WordPress source code still lives in an <a href="http://core.svn.wordpress.org">svn repository</a>, you can contribute patches without having to touch svn ever again.</p> <p>So, without further ado, here’s how you can generate and manage patches for WordPress Core (it assumes you’re comfortable with the command line).</p> <h3 id="1-find-a-ticket-to-work-on">1. Find a ticket to work on.</h3> <p>Patches need to be attached to tickets. You can browse through <a href="https://core.trac.wordpress.org/report">existing tickets</a> in trac or create a new one. The Contributor Handbook has more details on the <a href="http://make.wordpress.org/core/handbook/trac/">bug tracking workflow</a>.</p> <h3 id="2-clone-the-official-mirror">2. Clone the official mirror.</h3> <p><del>There’s an <a href="https://github.com/WordPress/WordPress">official mirror</a> of WordPress on github, so let’s use that:</del></p> <p>WordPress has an <a href="http://make.wordpress.org/core/2014/01/15/git-mirrors-for-wordpress/">official git mirror</a> which we can use:</p> <div class="highlighter-rouge"><div class="highlight"><pre class="highlight"><code>git clone git://develop.git.wordpress.org/ wordpress cd wordpress </code></pre></div></div> <h3 id="3-make-a-feature-branch">3. Make a feature branch.</h3> <p>Once you’ve got the ticket number, create a branch:</p> <div class="highlighter-rouge"><div class="highlight"><pre class="highlight"><code>git checkout master -b some-feature/123 </code></pre></div></div> <p>where 123 is the ticket number.</p> <p>Now, make the code changes you need to make and commit them (you can do several commits; it doesn’t matter):</p> <div class="highlighter-rouge"><div class="highlight"><pre class="highlight"><code>... edit some files... git commit -am "fixed the bug" ... edit some more files git commit -am "fixed edge case" </code></pre></div></div> <h3 id="3-generate-the-patch">3. Generate the patch.</h3> <p>Here’s the clever part: you can generate an <a href="http://scribu.net/wordpress/svn-patches-from-git.html" title="svn patches from git">svn-compatible patch</a> directly from git:</p> <div class="highlighter-rouge"><div class="highlight"><pre class="highlight"><code>git diff master... --no-prefix &gt; some-feature.123.diff </code></pre></div></div> <p>If you want to skip the <code class="highlighter-rouge">--no-prefix</code> arg, you can update your gitconfig:</p> <div class="highlighter-rouge"><div class="highlight"><pre class="highlight"><code>git config --global diff.noprefix true </code></pre></div></div> <p>All you have to now is upload <code class="highlighter-rouge">some-feature.123.diff</code> to the trac ticket. Done.</p> <h3 id="optional-publish-your-feature-branch">Optional: Publish your feature branch.</h3> <p>If it’s a big feature and you want to get feedback on it – without having to upload a new patch each time you make a change – you can publish your branch to <a href="http://github.com">Github</a>, <a href="https://bitbucket.org/">Bitbucket</a> or even your own server.</p> <p>So, for example, if I went ahead and created the scribu/WordPress repository on Github, all I’d have to do is tell my local clone about it:</p> <div class="highlighter-rouge"><div class="highlight"><pre class="highlight"><code>git remote add scribu git@github.com:scribu/WordPress.git </code></pre></div></div> <p>After that initial step is done, to publish my branch, I can just do:</p> <div class="highlighter-rouge"><div class="highlight"><pre class="highlight"><code>git push -u scribu some-feature/123 </code></pre></div></div> <h3 id="4-test-other-peoples-patches">4. Test other people’s patches.</h3> <p>It’s very likely that at some point you’ll encounter a ticket which already has a patch uploaded by someone else.</p> <p>First thing you need is an utility that makes it dead simple to download an attachment and apply it as a patch. Just add the following code to your <code class="highlighter-rouge">~/.bash_aliases</code> file:</p> <div class="highlighter-rouge"><div class="highlight"><pre class="highlight"><code># Apply a Trac patch. function tp() { curl "$1?format=raw" | patch -p0 } </code></pre></div></div> <p>And run <code class="highlighter-rouge">source ~/.bash_aliases</code>.</p> <p>Now, here’s how you use it:</p> <div class="highlighter-rouge"><div class="highlight"><pre class="highlight"><code>git checkout master -b revisions/23497 tp http://core.trac.wordpress.org/attachment/ticket/23497/23497.4.diff git add -A git commit -m "apply http://core.trac.wordpress.org/attachment/ticket/23497/23497.4.diff" </code></pre></div></div> <p>It gets tricky when a new version of the patch is uploaded and you have some additional commits in your feature branch.
